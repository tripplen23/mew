**An AI agent harness that extracts the behavioral DNA of software, then evolves a better implementation.**

Mew studies an existing system, turns observed behavior and user intent into an evidence-backed contract, and builds a new implementation against that contract. The source may be a repository, service, CLI, or an authorized application that Mew can operate through a browser or computer-use driver.

The output is not just generated code. A Mew run is expected to produce a pinned baseline, evidence, an approved contract, reviewable implementation slices, differential verification, known deviations, and a handoff report.

## Why Mew

Vibe coding makes prototypes cheap. It does not make the next step cheap.

A prototype may validate the product while carrying the wrong language, framework, architecture, design, or deployment model for production. Replacing it safely requires more than translating files. Someone must discover what the system actually does, decide what should survive, and prove that the replacement still behaves as intended.

Mew calls that reusable description the system's **behavioral DNA**.

## Example journeys

### Evolve a runtime

```text
Input:
  Repository: an existing AI inference service
  Intent: replace measured Python bottlenecks with a production-suitable
          Rust implementation

Mew:
  pins and reproduces the source -> profiles the real execution path
  -> extracts API, fixture, numerical, error, and performance contracts
  -> asks the user to approve them -> migrates in slices
  -> runs baseline and candidate on the same scenarios
  -> returns a PR with parity and benchmark evidence
```

The requested rewrite is treated as a hypothesis. If measurements do not support it, Mew should say so.

### Reconstruct an application with a new identity

```text
Input:
  URL: https://nguyenducbinh.vercel.app/
  Intent: preserve the portfolio's content and important interactions,
          then rebuild it with a stronger manga visual direction

Mew:
  maps public routes -> exercises links and controls in a real browser
  -> captures responsive, network, console, DOM, and accessibility behavior
  -> drafts a behavior, content, and visual-direction contract
  -> asks the user to approve it -> builds the new application
  -> replays the journeys and verifies the result
```

Mew should preserve the approved product behavior, not blindly copy pixels or protected assets. Authorization, provenance, and publication remain explicit boundaries.

## How it works

```text
intake
  -> acquire and reproduce
  -> observe
  -> extract behavioral DNA
  -> user approval
  -> plan the evolution
  -> implement in checkpoints
  -> compare baseline and candidate
  -> hand off code and evidence
```

Mew owns durable state, policy, workspaces, evidence, checkpoints, and verification. Models and skills handle reasoning that is still ambiguous. Stable workflows can graduate from skills into native runtime capabilities.

## Current status

Mew has a working local coding-agent foundation:

- Rust protocol, engine, server, and Ratatui client;
- Rig-based model and tool loop;
- filesystem, search, shell, memory, and skill tools;
- plan/build mode gating;
- conversation persistence and SSE streaming;
- OpenTelemetry/Langfuse tracing;
- Go MCP adapter using the official Model Context Protocol SDK.

The migration-run, sandbox, behavioral-contract, differential-verification, browser, and computer-use layers are on the roadmap. Mew is experimental and should not be trusted with production or irreversible actions without human review.

## Repository layout

This repository contains multiple components under one root:

```text
mew/
  mew-core/   Rust workspace: protocol, engine, server, TUI client
  mew-mcp/    Go MCP server: exposes Mew to external AI agents
  docs/       Product and architecture documents
  .github/    Rust and Go CI workflows
```

### `mew-core`

```text
mew-core/
  .mewcode/  component-local project skills
  crates/
    protocol/  shared types and wire format, with no I/O
    engine/    Rig-based harness, tools, skills, streaming, and memory
    server/    Axum backend, APIs, and persistent stores
    client/    Ratatui terminal UI and CLI dispatcher
```

### `mew-mcp`

`mew-mcp` is a thin stdio MCP adapter. It forwards health, chat, and session operations to `mewcode-server` over HTTP and SSE. It does not embed or duplicate the Rust engine.

## Getting started

```bash
cp .env.example .env
# Set OPENCODE_GO_API_KEY to an OpenCode Go or compatible endpoint key.

cd mew-core
cargo run -p mewcode-server &
cargo run -p mewcode-client -- tui
```

The Rust workspace requires Rust 1.85 or newer and edition 2024 support.

To build the MCP adapter:

```bash
cd mew-mcp
go build ./cmd/mew-mcp
```

## Design principles

- Evidence before implementation.
- User-approved intent over agent inference.
- Baseline and candidate remain separate.
- Structured artifacts over transcript-only state.
- Official SDKs and documented standards before custom protocols.
- Comparison against behavior, not source-code resemblance.
- Safe pause and honest blockers instead of fabricated success.
- No merge, deployment, publication, or irreversible action without approval.
